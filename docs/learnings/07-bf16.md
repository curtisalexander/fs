# Learning 07 — bf16: the weights' number format (and why we widen lazily)

> **Date:** 2026-07-06 · **Context:** M1, the dtype in `model.safetensors` · **Status:** living
>
> 📖 *Inference Engineering* §5.1 (numeric formats / quantization neighborhood, p.120)
> 🔧 `ds4`: `f16`/`f32` tensor handling in `ds4.c`; `metal/*` compute in `f32`
> 🧭 Google Brain's bfloat16; Raschka on mixed-precision inference

Qwen3-0.6B's weights are **all `BF16`** (see [`learning 01`](01-safetensors-vs-gguf.md)).
Before we load a single tensor it's worth knowing exactly what those two bytes
*are* — because M1's signature design decision, **"lazy bf16"**, only makes sense
once you can see the format. This note is that format, bit by bit.

This is *not* quantization (that's M5, and it's a different, lossy story). bf16 is
a plain IEEE-style float, just a narrow one.

---

## Three floats, same idea, different budgets

A binary float spends its bits on three fields: **sign · exponent · mantissa**
(the fraction). The exponent sets the *range* (how big/small), the mantissa sets
the *precision* (how many significant bits). Here are the three formats we care
about, drawn to their bit widths:

```
          sign  exponent                mantissa (fraction)
 fp32  =  [ 1 ][ 8: eeeeeeee ][ 23: mmmmmmmmmmmmmmmmmmmmmmm ]   32 bits
 fp16  =  [ 1 ][ 5: eeeee    ][ 10: mmmmmmmmmm             ]   16 bits
 bf16  =  [ 1 ][ 8: eeeeeeee ][  7: mmmmmmm                ]   16 bits
             │        │                    │
           +/-     RANGE                PRECISION
```

The key comparison is **bf16 vs fp16** — both are 16 bits, but they split the
budget differently:

| format | bits | exponent | mantissa | range | precision |
|--------|-----:|---------:|---------:|-------|-----------|
| fp32   | 32   | 8        | 23       | huge  | high      |
| fp16   | 16   | **5**    | **10**   | small | higher (of the two) |
| bf16   | 16   | **8**    | **7**    | **= fp32** | lower |

**bf16 keeps fp32's exponent (8 bits) and throws away mantissa instead.** So bf16
covers the *same dynamic range* as fp32 (≈ 10^±38) — it just represents each value
more coarsely (7 fraction bits ≈ 2–3 decimal digits). fp16, by keeping only 5
exponent bits, tops out around 65504 and underflows early, which is why training/
inference in fp16 famously needs loss-scaling gymnastics to avoid overflow.
**bf16 trades precision for range, and for neural nets range is the thing that
keeps you numerically safe.** That's why modern weights ship as bf16.

## The punchline: bf16 *is* the top 16 bits of an fp32

Line the two up. Because bf16 and fp32 share the **same sign width and the same
8-bit exponent**, a bf16 value is *exactly* an fp32 value with its low 16 mantissa
bits chopped off:

```
 fp32:  s eeeeeeee mmmmmmm|mmmmmmmmmmmmmmmm      (23 mantissa bits)
 bf16:  s eeeeeeee mmmmmmm                       (keep the top 7, drop the low 16)
        └───────── same 16 bits ─────────┘
```

So **widening bf16 → f32 is just a 16-bit left shift** — put the bf16 bits back in
the high half and zero-fill the low half:

```rust
pub fn bf16_to_f32(bytes: [u8; 2]) -> f32 {
    f32::from_bits((u16::from_le_bytes(bytes) as u32) << 16)
}
```

No lookup table, no branch, no special-casing. And it's **exact for every bf16
value** — normals, subnormals, and `inf`/`NaN` all round-trip, because we never
touch the sign or exponent, only re-seat the mantissa in a wider field. (Rounding
only enters the *other* direction — fp32 → bf16 — which we never do here.)

> Contrast with M5's quantization: a `Q4_K` weight is **not** a float you can shift
> into place — it's a 4-bit index into a block that shares a scale, and decoding it
> is real arithmetic. bf16 → f32 is free; dequant is not. Keep the two mentally
> separate.

## Why M1 keeps the bytes raw (the "lazy bf16" decision)

We *have* `bf16_to_f32`, but M1 **never calls it.** Here's the arithmetic that
forces that choice:

```
 model.safetensors  =  ~1.4 GB of bf16   (2 bytes / weight)
 eager convert to f32  →  ~2.8 GB          (4 bytes / weight)
 ─────────────────────────────────────────────────────────
 eager conversion DOUBLES resident memory and copies every weight up front —
 which is exactly what mmap was supposed to save us from (learning 06).
```

So in M1:

- tensors stay **raw bf16 bytes** as `&[u8]` slices into the mmap (zero-copy);
- our checksums run over those **raw bytes** — proving we read the right region of
  the file, not that we can do float math;
- `bf16_to_f32` is written and unit-tested, but **first *called* at M2**, one
  value at a time, only for the weights a given matmul actually touches.

This is the "learning-first, then fast" rule in miniature: the *correct* and
*clear* thing is to convert at the point of use; converting eagerly would be a
premature (and memory-doubling) optimization in the wrong direction.

---

## Mental model to keep

> **bf16 = fp32 with a haircut.** Same range, coarser steps, half the bytes. To
> widen, shift it back into fp32's high half — free and exact. To *narrow* (which
> we don't), you'd round, and rounding is where number formats get interesting.
> Quantization (M5) is a different animal: the bytes need *decoding*, not shifting.
